아티클 개요

* Andrew Chan이 2012년 Nielsen의 실험을 2025년 하드웨어로 재현한 프로젝트
* 25.5시간, $462로 약 10억 개 웹페이지 크롤링
* 현대 환경에서 대규모 크롤링의 병목이 어디로 이동했는지 분석



1. 가장 인상적이었던 내용

| 예상했던 병목과 실제 병목이 달랐다는 점

* 예상: 네트워크 속도, DNS 처리
* 실제: CPU (네트워크·DNS는 여유 있었음)
* 원인: HTML 파싱 + SSL 핸드셰이크 (SSL만 CPU 시간의 약 25%)
* 대응: 파서 lxml → selectolax 교체, HTML 250KB로 truncate



2. 아키텍처 분석

구조: 수집·파싱·저장을 분리하지 않고, 모든 기능을 담은 독립 노드 12개로 구성 (노드 간 통신 없음, 도메인별 샤딩)

컴포넌트
	역할
	특징

Redis
	URL·크롤링 상태 관리
	frontier, bloom filter

Fetcher
	웹페이지 다운로드
	IO-bound → asyncio

Parser
	링크 추출
	CPU-bound → 별도 프로세스

Storage
	콘텐츠 저장
	노드별 NVMe SSD


장점
* 구조 단순, 노드 간 장애 전파 없음

한계
* Wikipedia·Yahoo처럼 링크 많은 도메인 → Hot Shard 문제 → 특정 노드 메모리 부족으로 다운



3. 어떤 의사결정이 놀라웠나요?

S3 대신 EC2 로컬 NVMe 저장소를 선택한 것
이유 - S3 사용 시 하루 저장 비용 약 $5,183 (전체 크롤링 비용의 10배 이상) 

정확성보다 속도·비용 효율을 택한 결정들

* Bloom Filter 오탐으로 인한 URL 누락 허용
* 큰 HTML은 250KB까지만 처리
* JavaScript 실행 없이 HTML만 수집

이 세 가지 모두 "일부 데이터 손실 vs 속도·비용" 트레이드오프에서 후자를 선택
