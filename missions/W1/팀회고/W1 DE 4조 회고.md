## W1M1. 데이터셋의 경제적 가치 & 변수 간 상관관계 분석
### 공통적으로 언급된 부분

경제적 가치 측면에서는 데이터 분석 결과가 실제 비즈니스 의사결정에 활용될 수 있다는 점에 팀원들의 의견이 모였다.

* 제조사 관점에서는 차량의 연비, 마력, 배기량 등의 데이터를 분석하여 R&D 방향을 설정하고 불필요한 제조 비용과 리소스 낭비를 줄일 수 있다는 의견이 나왔다.
* 소비자·유통 관점에서는 적정 가격 산정, 중고차 시세 예측, 렌터카 운영 최적화 등 다양한 비즈니스 분야에 활용할 수 있다는 의견이 많았다.

상관관계 분석에서도 팀원들의 해석은 큰 차이가 없었다.

![alt text](corr_plot.png)

* wt(무게)와 mpg(연비)는 강한 음의 상관관계를 보여 차량이 무거울수록 연비가 낮아지는 경향을 확인하였다.
* cyl(실린더 수), wt(무게), disp(배기량)은 강한 양의 상관관계를 보여 실린더 수와 무게가 증가할수록 배기량도 함께 증가하는 경향을 확인하였다.

이를 종합하면 배기량이 증가할수록 차량의 무게도 증가하고, 무게 증가는 다시 연비 저하로 이어지는 관계임을 확인할 수 있었다.

### 다르게 언급된 부분

경제적 가치를 설명하는 과정에서 강조한 활용 사례에는 차이가 있었다.

제조사의 R&D 효율화와 차세대 차량 설계에 초점을 맞춘 팀원이 있었고, 보험료 산정이나 마케팅 세분화 등 다양한 산업 분야에서의 활용 가능성을 중심으로 설명한 팀원도 있었다. 목표 연비를 달성하기 위해 차량의 무게와 실린더 수를 어떻게 설계해야 하는지를 예시로 들어 설명한 팀원도 있었다.

상관관계 분석에서도 중점적으로 살펴본 변수는 조금씩 달랐다. cyl-disp, wt-disp를 중심으로 분석한 팀원이 있는 반면, wt-mpg 관계를 핵심으로 해석한 팀원도 있었다. 다만 최종적인 해석은 모두 비슷한 방향이었다.

### 논의 결과 및 결론

하나의 데이터셋만으로도 제조, 유통, 금융(보험), 마케팅 등 다양한 분야에서 활용 가능한 가치를 도출할 수 있다는 데 의견이 모였다. 또한 wt, disp, mpg의 관계를 종합하면 성능 향상을 위해 배기량을 늘릴 경우 무게가 증가하고, 이는 연비 저하로 이어질 가능성이 크다. 따라서 차량을 설계할 때는 성능뿐만 아니라 경량화와 엔진 효율도 함께 고려해야 한다는 결론을 내렸다.

## W1M2. SQLite를 활용한 SQL 실습

### 공통적으로 언급된 부분

팀원 모두 Python의 sqlite3 모듈을 활용해 데이터베이스를 연결하고 SQL 쿼리를 실행하는 과정을 실습하며 SQLite의 기본 사용 방법을 익혔다.

또한 SQL Tutorial에서 다루는 ANY, ALL, SELECT INTO, Stored Procedure 등의 문법이 SQLite에서는 지원되지 않는다는 점을 공통적으로 확인하였다. 이를 통해 DBMS마다 지원하는 SQL 문법과 기능에 차이가 있다는 점을 알게 되었다.

### 다르게 언급된 부분

학습한 내용의 깊이와 초점은 팀원마다 조금씩 달랐다.

한 팀원은 SQLite에서 일부 문법을 사용할 수 없었던 점은 아쉬웠지만, 평소 자주 접하지 못했던 ANY, ALL, IN 등의 문법을 이론적으로 학습할 수 있었다는 점을 긍정적으로 평가하였다.

다른 팀원은 한 단계 더 나아가 EXISTS와 JOIN의 차이, UNION과 UNION ALL의 차이, 상관 서브쿼리의 동작 방식까지 추가로 정리하며 각 문법의 특징과 활용 방법을 함께 학습하였다.

### 논의 결과 및 결론

팀에서는 단순히 SQLite에서 지원 여부만 확인하는 데 그치지 않고, 문법 차이가 발생하는 이유와 각 문법이 어떤 상황에서 사용되는지까지 함께 이해하는 것이 중요하다는 의견이 나왔다. SQLite는 학습과 간단한 프로젝트에는 적합하지만, 실무에서는 사용하는 DBMS의 지원 문법을 미리 확인하는 습관이 필요하다는 결론을 내렸다.

## W1M3. IMF 데이터 수집 방식 & ETL 버전 관리

### 공통적으로 언급된 부분

Wikipedia를 스크래핑하는 방식보다 IMF에서 제공하는 공식 데이터를 활용하는 것이 더 적절하다는 데 의견이 모였다. 공식 API나 데이터 파일을 활용하면 HTML 구조 변경의 영향을 받지 않아 데이터의 안정성과 신뢰성을 높일 수 있기 때문이다.

또한 데이터를 갱신할 때 기존 데이터를 덮어쓰지 않고 과거 데이터를 함께 보관해야 한다는 점에도 모두 동의하였다. 시계열 분석이나 과거 데이터 비교를 위해서는 이전 데이터가 유지되어야 한다는 의견이었다.

ETL 과정에서도 Load 단계는 replace 방식이 아닌 append 방식을 사용하는 것이 적절하다는 의견으로 정리되었다.

### 다르게 언급된 부분

데이터를 수집하는 방법은 팀원마다 조금씩 달랐다.

* requests를 이용해 REST API의 JSON 데이터를 수집하는 방법
* IMF에서 제공하는 Excel 또는 CSV 파일을 활용하는 방법
* Wikipedia API를 활용하는 방법

과거 데이터를 관리하는 방식도 조금씩 차이가 있었다.

* Release_Version과 같은 컬럼을 추가하여 버전을 관리하는 방법
* 파일명이나 테이블명을 다르게 저장하는 방법
* v1, v2와 같이 버전 번호를 부여하는 방법

구현 방식에는 차이가 있었지만 데이터를 저장할 때 기준 시점을 함께 기록해야 한다는 점은 모두 동일한 의견이었다.

### 논의 결과 및 결론

팀에서는 IMF의 공식 API 또는 Excel·CSV 데이터를 활용하여 데이터 수집의 안정성을 높이기로 하였다. 또한 ETL의 Load 단계에서는 append 방식을 사용하고 Release_Version이나 수집일자와 같은 버전 정보를 함께 저장하여 과거 데이터를 관리하기로 하였다. 조회 시에는 최신 데이터를 기본으로 사용하되, 필요에 따라 과거 버전과 비교할 수 있도록 SQL 쿼리를 작성하는 방향으로 의견을 정리하였다.
