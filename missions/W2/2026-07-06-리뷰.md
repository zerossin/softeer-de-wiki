# 2026-07-06 리뷰

오전에는 데이터 엔지니어링 수업을 들었다.

이번 수업에서는 데이터를 처리할 때 단순히 기능을 구현하는 것이 아니라, 데이터 품질과 재사용성을 고려하는 설계의 중요성을 배웠다. 컨텍스트를 기반으로 데이터를 이해해야 한다는 점, 하드코딩 대신 설정(Configuration)을 분리하는 방법, 캡슐화와 수도 코드를 활용한 설계 방법 등을 학습했다. 또한 데이터 품질(Veracity), 캐시 활용, 데이터 병렬 처리, 멀티프로세싱과 멀티스레딩의 차이, GIL, 스케일 업과 스케일 아웃, Docker와 가상화, 프로토타이핑과 프로덕트 개발 등 데이터 엔지니어링 전반에 대한 다양한 개념을 접할 수 있었다.

오후에는 멀티프로세싱 과제를 수행했다. 당장 코드로는 낯선 내용이었던 만큼 초반에는 AI의 도움을 받아 코드를 작성한 뒤, 생성된 코드를 직접 분석하며 문법과 동작 원리, 관련 개념을 하나씩 이해하는 데 집중했다.

Pool, Process, Queue 예제를 순서대로 수행한 뒤, 이를 종합하는 멀티프로세스 파이프라인 구현 과제를 진행했다. w2m4 과제에서는 앞선 예제에서 학습한 내용을 바탕으로 AI의 도움 없이 직접 구현을 시도했다.

w2m4 과제를 수행하는 과정에서 코드 구현뿐만 아니라 다양한 실험도 함께 진행했다. `multiprocessing.Queue()`를 사용하는 이유를 고민하며 `deque`와의 차이를 비교했고, 프로세스 간 메모리 공간이 분리되어 있기 때문에 별도의 프로세스 간 통신용 큐가 필요하다는 점을 확인했다.

또한 `get_nowait()`를 사용할 때 `Queue.empty()`만으로는 안전하게 종료 여부를 판단하기 어렵다는 점을 직접 확인했다. 여러 프로세스가 동시에 큐에 접근하는 환경에서는 `empty()`가 `False`를 반환하더라도 다른 프로세스가 먼저 데이터를 가져가 `queue.Empty` 예외가 발생할 수 있음을 실험을 통해 확인했다.

결과 저장 방식도 개선해 보았다. 처음에는 완료 메시지 전체 문자열을 큐에 저장했지만, 이후에는 작업 번호와 프로세스 이름만 저장하고 출력 시 문자열을 조합하도록 변경하여 불필요한 데이터 저장을 줄였다.

추가적으로 작업 개수를 늘려 실험하던 중 특정 시점부터 하나의 프로세스가 대부분의 작업을 처리하는 현상을 관찰했다. 원인을 조사하는 과정에서 `multiprocessing.Queue()`가 내부적으로 백그라운드 스레드를 통해 데이터를 파이프로 전달하는 구조를 사용한다는 내용을 확인했고, `get(timeout=1)`을 적용했을 때 작업 분배가 보다 균등하게 이루어지는 것도 확인했다.

마지막으로 멀티프로세스 큐의 동작 방식이 운영체제에서 학습했던 동기화 개념과 관련이 있을 것이라는 생각이 들었고, 세마포어와의 연관성에 대해서도 추가로 찾아볼 필요성을 느꼈다.
